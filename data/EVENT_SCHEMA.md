# Event Schema

This document defines the event object shape consumed by the frontend in `app.js` and produced by the Engage scraper in `scripts/scrape-engage.mjs`.

Use this document before editing `data/events.json` or changing scraper output.

## Event Object

Every event is represented as a JSON object inside an array.

Current normalized shape:

```json
{
  "id": "seed-1",
  "title": "Free Coffee & Donuts",
  "host": "Student Affairs",
  "campus": "Norman",
  "location": "Bizzell Memorial Library, main entrance",
  "description": "Grab a free coffee and donut to kickstart your morning.",
  "category": "Breakfast",
  "dietary": "Vegetarian options",
  "link": "",
  "start": "2025-09-23T08:30:00-05:00",
  "end": "2025-09-23T10:00:00-05:00",
  "createdAt": "2025-09-20T12:00:00-05:00"
}
```

## Field Reference

| Field | Required for committed data | Type | Notes |
|------|------------------------------|------|-------|
| `id` | Recommended | string | Should be stable across reloads. If omitted, the frontend generates one at runtime. |
| `title` | Yes | string | Trimmed by the frontend. Empty or missing titles become `Untitled`. |
| `host` | No | string | Organization or source name. Empty string is allowed. |
| `campus` | Recommended | string | Current UI options are `Norman`, `OUHSC`, `Tulsa`, and `Online`. Empty string is allowed but weakens filtering. |
| `location` | Recommended | string | Human-readable location. Empty string is allowed but not ideal. |
| `description` | No | string | Freeform event details. |
| `category` | Recommended | string | Current UI options are `Breakfast`, `Lunch`, `Dinner`, `Snacks`, `Pizza`, `Pantry`, `Giveaway`, and `Workshop`. |
| `dietary` | No | string | Dietary notes such as vegetarian or vegan options. |
| `link` | No | string | Event URL. Use `""` if none. |
| `start` | Yes | string | Must be a date string that JavaScript `Date` can parse. Prefer ISO 8601 with timezone offset. |
| `end` | No | string or `null` | If present, it must also be parseable by JavaScript `Date`. Prefer ISO 8601 with timezone offset. |
| `createdAt` | Recommended | string | Creation or ingestion timestamp. If omitted, the frontend fills in the current time at load. |

## Safe Authoring Rules

Follow these rules for anything committed to `data/events.json`:

- Always include `title` and `start`.
- Use ISO 8601 date strings with an explicit timezone offset when possible.
- Keep `id` stable once committed so the same event does not change identity between deploys.
- Prefer the existing campus and category vocabulary so filters continue to work.
- Use empty strings instead of missing keys for optional text fields when editing manually.

## Source-Specific Expectations

### Seeded Events In `data/events.json`

Seeded events should be the cleanest source in the project.

Expectations:
- Include every field in the normalized shape when practical
- Use stable IDs such as `seed-1`, `seed-2`, or another deterministic naming pattern
- Use explicit timezone offsets in `start`, `end`, and `createdAt`
- Keep campus and category aligned with the current UI filter values

### Scraped Engage Events In `data/engage.json`

The scraper currently emits the same normalized shape used by the frontend.

Current scraper conventions:
- `id` is generated as `engage-` plus a hash of the event URL
- `category` is inferred from food signals and falls back to `Giveaway`
- `dietary` defaults to `""`
- `campus` is inferred from location, title, and description text when possible
- `createdAt` is set to the scrape time
- discovery currently uses the public OU Engage RSS feed as the primary source

Because this source is heuristic, fields like `campus`, `location`, `description`, and `end` may be incomplete.

The scraper also writes `data/engage-meta.json` with crawl metadata. That sidecar file is operational metadata, not an event array.

### Local Browser Submissions

Local submissions are created by the form in `index.html` and stored in browser `localStorage`.

Current form behavior:
- `title`, `location`, `campus`, `category`, and `start` are required by the form
- `id` is generated in the browser
- `createdAt` is generated in the browser
- `end`, `description`, `link`, `dietary`, and `host` are optional

These events are not committed to the repo.

### Private Email-Derived Events In `data/email-events.json`

This is an optional local file generated by `scripts/import-email-events.mjs`.

Current importer conventions:
- `source` is set to `email`
- `id` is generated from the filename, title, and extracted start date
- discovery comes from text files placed in `data/email-raw/`
- only likely food-related events with an extracted start time are emitted
- generated files are intended to stay local and are ignored by git

The importer also writes `data/email-meta.json` with counts and parsing errors.

## Current Frontend Normalization Behavior

`normalizeEvents()` in `app.js` currently applies these rules:

- Non-array input becomes an empty array
- `id` is generated if missing
- `title` is trimmed and defaults to `Untitled`
- `host` is trimmed and defaults to `""`
- Optional text fields default to `""`
- `createdAt` defaults to the current time if missing
- Records with a missing `start` field are dropped

Important caveat:
- The current code assumes any provided `start`, `end`, or `createdAt` value is parseable by JavaScript `Date`
- A truthy but malformed date string can throw during normalization

For now, treat malformed dates as invalid input and do not commit them.

## Invalid Record Handling

Until validation is hardened in a later ticket, contributors should use these rules:

- Reject any record missing `title` or `start`
- Reject any record whose `start`, `end`, or `createdAt` cannot be parsed by JavaScript `Date`
- Reject any record where `end` is intentionally present but not a real timestamp
- Avoid unknown campus or category values unless the UI is updated at the same time

Planned follow-up work:
- make invalid-date handling explicit in the frontend
- add stronger deduplication across sources
- add tests for normalization and filtering behavior

```json
{
  "id": "seed-4",
  "title": "Free Bagel Study Break",
  "host": "Engineering Student Council",
  "campus": "Norman",
  "location": "Sarkeys Energy Center, atrium",
  "description": "Bagels and coffee while supplies last.",
  "category": "Breakfast",
  "dietary": "Vegetarian options",
  "link": "https://example.com/study-break",
  "start": "2026-04-09T09:00:00-05:00",
  "end": "2026-04-09T11:00:00-05:00",
  "createdAt": "2026-04-08T12:00:00-05:00"
}
```
